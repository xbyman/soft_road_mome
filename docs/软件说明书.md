# 软件说明书

## 1. 软件概述

`road_mome_demo_v1` 是基于 Road-MoME 模型构建的单机浏览器演示平台，用于在本地环境下完成演示样本的单帧检测、结果展示与结果导出。软件采用本地离线运行方式，通过浏览器页面提供统一交互界面。
## 2. 软件目标与定位

### 2.1 软件目标

本软件的主要目标如下：

- 提供一个可直接运行的本地浏览器演示界面。
- 支持对预置演示样本进行单帧检测与结果展示。
- 支持对检测结果进行图像、结构化结果和摘要信息导出。
- 在异常输入或资源缺失情况下，能够进行启动前校验和运行前校验，降低演示过程中页面直接崩溃的风险。

### 2.2 软件定位

本软件定位为：

- Road-MoME 单机浏览器演示平台 V1.0
- 面向教师、评审和后续维护使用者的展示与说明软件
- 面向本地离线样本的功能验证平台

本软件不定位为：

- 商用系统
- 实时车载部署系统
- 在线服务化平台
- 多用户并发访问系统

## 3. 运行环境要求

### 3.1 基本运行环境

| 项目 | 要求说明 |
| --- | --- |
| 操作系统 | Windows 本地环境 |
| Python | 已验证环境为 Python 3.13.1 |
| 运行方式 | 本地单机离线运行 |
| 页面访问方式 | 本机浏览器访问本地服务 |

### 3.2 主要依赖环境

软件当前实现依赖以下类别组件：

- Dash：用于构建浏览器页面与交互回调
- NumPy：用于样本数组与推理结果处理
- PyYAML：用于读取配置文件
- Pillow：用于结果图像导出
- PyTorch 及相关模型依赖：用于离线推理执行

项目依赖清单以 `requirements_runtime.txt` 为准。

### 3.3 本地资源要求

软件启动和运行需要以下本地资源完整存在：

- 配置文件目录：`config/`
- 模型权重目录：`weights/`
- 演示数据目录：`demo_data/`
- 输出目录：`outputs/`

其中演示数据应至少包含：

- `demo_data/index.json`
- `demo_data/jpg/`
- `demo_data/npz/`

## 4. 软件目录与组成

### 4.1 项目目录结构

| 目录或文件 | 作用说明 |
| --- | --- |
| `app.py` | 软件主入口，负责启动本地页面服务 |
| `app_bootstrap.py` | 运行时初始化与上下文组装 |
| `ui/` | 页面布局、交互回调、显示格式化与图像展示辅助 |
| `runtime/` | 推理、可视化、导出、校验、日志等运行时能力 |
| `data_access/` | 演示索引、图像和样本数据读取 |
| `models/` | Road-MoME 模型结构相关实现 |
| `config/` | 配置文件目录 |
| `weights/` | 模型权重文件目录 |
| `demo_data/` | 演示样本数据目录 |
| `outputs/` | 结果图像、日志和导出文件目录 |
| `docs/` | 说明文档目录 |

### 4.2 软件组成说明

从功能组成上看，当前软件可划分为以下几个部分：

1. 应用入口层  
负责启动 Dash 页面应用，并完成各模块装配。

2. 页面交互层  
负责样本选择、阈值设置、运行按钮响应、状态区与结果区展示。

3. 运行时处理层  
负责启动校验、样本校验、模型推理、可视化结果生成、导出和日志记录。

4. 数据访问层  
负责从本地演示目录读取 `index.json`、图像文件和 `.npz` 样本特征文件。

5. 模型层  
负责离线推理所需的模型结构定义与权重加载支持。

## 5. 主要功能说明

### 5.1 页面启动

软件启动后可在本机浏览器中打开本地页面，提供统一的单页演示界面，用于后续样本选择、运行检测和结果查看。

### 5.2 演示样本选择

系统从本地 `demo_data/index.json` 读取可用样本列表，并在页面中以下拉框形式提供选择。当前版本基于预置演示样本进行单帧演示，不支持用户自行上传样本。

### 5.3 阈值设置

页面提供阈值设置能力，用于在单帧检测时调整当前运行所使用的阈值参数。阈值由页面控件传入推理流程，并参与结果统计与导出记录。

### 5.4 单帧检测运行

用户选择样本后，可触发单帧检测流程。系统完成样本读取、运行前校验、模型推理和结果汇总，并在页面中返回对应结果。

### 5.5 四宫格结果显示

推理完成后，系统生成四宫格可视化结果，并在页面结果区中显示。该结果用于直观展示当前样本的检测与可视化输出。

### 5.6 统计信息显示

页面提供与当前帧结果相关的统计信息显示，用于展示本次单帧运行的结果摘要，便于答辩说明与结果观察。

### 5.7 运行并导出结果

除仅运行检测外，系统还支持“运行并导出”流程。在完成单帧推理后，可同步导出以下结果：

- 可视化结果图像
- 单帧结果 JSON 文件
- 结果摘要 CSV 文件

### 5.8 启动前校验

系统在启动阶段执行 `startup validation`，用于检查以下内容：

- 配置文件是否存在且可解析
- 权重文件是否存在
- 演示数据目录是否完整
- `index.json` 是否存在且结构合法
- 索引中引用的 `jpg` 和 `npz` 路径是否有效
- 输出目录是否可创建、可写

### 5.9 样本运行前校验

系统在读取样本后、进入推理前执行 `runtime sample validation`，用于检查：

- 样本关键字段是否存在
- 图像数据是否为空
- 数组维度与形状是否符合要求
- `meta[:,2]` 是否存在有效 patch

### 5.10 日志记录

系统具备本地日志记录能力，用于记录启动、校验、推理、导出和异常信息，便于问题定位和运行留痕。

## 6. 软件启动方式

### 6.1 启动入口

软件主入口为：

- `app.py`

在项目根目录下执行：

```

```text

```

启动过程中，系统会先完成运行时初始化和启动前校验；校验通过后，再初始化样本提供器、推理引擎和导出器，并加载页面所需初始状态。若启动校验失败，系统会给出错误提示并阻止进入不可靠运行状态。

## 7. 输入与输出说明

### 7.1 输入说明

## 8. 异常处理与提示机制

### 8.1 异常处理原则

当前版本遵循以下异常处理原则：

- 优先在启动阶段和运行前阶段拦截问题
- 页面提示以简洁、可理解为主
- 避免页面因单个样本异常直接崩溃
- 保留日志以支持后续问题定位

### 8.2 启动阶段异常处理

当配置文件缺失、权重路径错误、演示数据目录不完整、索引为空或输出目录不可写时，系统会在启动前校验阶段报告异常，并阻止进入后续不可靠运行流程。

### 8.3 运行阶段异常处理

当某个样本字段缺失、数组形状异常或 `meta[:,2]` 全 0 时，系统会在样本运行前校验阶段拦截该样本，不进入推理流程，并在页面状态区给出对应提示。

### 8.4 提示与日志机制

异常信息主要通过两种方式呈现：

- 页面状态区提示
- 本地日志文件记录

这种机制兼顾了演示时的可读性和工程上的可追踪性。
