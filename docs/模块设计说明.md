# 模块设计说明

## 1. 文档目的与适用范围

本文档用于说明 `road_mome_demo_v1` 单机浏览器演示平台的模块划分、职责边界、数据流转关系及异常处理机制.

本文档面向当前已实现版本，重点描述以下内容：

- 软件总体架构
- 目录结构与模块职责
- 单帧检测演示的数据流
- 输入输出设计
- 校验、异常处理与日志机制
- 当前版本功能边界

---

## 2. 软件总体架构

本系统定位为一套**单机浏览器演示平台**。系统以 Dash 为前端承载形式，在本地浏览器中提供单页式交互界面，完成演示样本选择、单帧推理、结果可视化、结果导出及状态展示。

从结构上看，系统采用分层设计，主要包括：

1. 应用入口层  
   负责启动 Dash 应用、装配运行时上下文、挂载布局并注册回调。

2. UI 层  
   负责页面布局、交互回调、显示格式化与图像展示辅助。

3. 运行时层  
   负责模型权重加载、推理执行、结果可视化、结果导出、启动校验和日志记录。

4. 数据访问层  
   负责演示索引读取、图像加载、特征文件加载以及样本对象组装。

5. 模型层  
   负责 MoME 模型结构及其主干网络定义。

6. 配置与资源层  
   负责系统配置、演示样本资源、模型权重和输出目录组织。

当前系统主运行链可概括为：

`sample_provider -> infer_engine -> visualizer -> exporter -> app`

其中，应用界面负责调度与展示，运行时模块负责核心计算与持久化。

## 4. 核心模块设计

### 4.1 应用入口层

#### 4.1.1 `app.py`

`app.py` 是系统的最薄入口文件，主要职责包括：

- 创建 Dash 应用实例
- 调用 `app_bootstrap.py` 完成运行时初始化
- 挂载页面布局
- 注册回调函数
- 启动本地 Web 服务

该文件不承担业务处理逻辑，不直接执行推理、导出或校验，仅负责系统启动与装配。

#### 4.1.2 `app_bootstrap.py`

该文件负责应用初始化和运行时上下文组装，主要职责包括：

- 调用 `runtime/validators.py` 执行启动前校验
- 初始化 `LocalSampleProvider`
- 初始化 `MomeInferEngine`
- 初始化 `ResultExporter`
- 汇总初始化消息、错误信息、样本列表、权重路径、默认阈值等状态
- 组装 `RuntimeContext`，供 UI 层共享使用

该模块是应用入口层与运行时层之间的桥接点。

---

### 4.2 UI 层

#### 4.2.1 `ui/layout.py`

该模块负责页面静态布局与可视组件构建，包括：

- 页面头部区域
- 左侧控制区
- 右侧结果显示区
- 下方统计信息区
- 状态信息区
- 各类占位提示组件

当前版本已采用单页仪表盘式布局，强调“控制区 + 主结果区 + 信息区”的组织方式。

#### 4.2.2 `ui/callbacks.py`

该模块负责所有 Dash 回调注册，主要完成：

- 阈值显示文字更新
- “运行检测”按钮响应
- “运行并导出”按钮响应
- 单帧样本读取、运行前样本校验、推理、可视化、导出流程串联
- 结果区、统计区、状态区回写

该模块是 UI 层和运行时层之间的主要交互入口。

#### 4.2.3 `ui/formatters.py`

该模块用于处理界面显示相关的轻量格式化逻辑，包括：

- 数值格式化
- 整数格式化
- 阈值文字格式化
- 统计项提取
- 导出路径文本拼接

该模块不包含推理逻辑，仅服务于界面显示。

#### 4.2.4 `ui/image_utils.py`

该模块用于处理页面图像展示相关功能，包括：

- 将 RGB NumPy 图像编码为 base64 data URL
- 构建 Dash 可直接展示的图像组件

该模块实现了“无需依赖临时文件即可在页面显示结果图”的设计目标。

---

### 4.3 运行时层

#### 4.3.1 `runtime/infer_engine.py`

该模块负责模型推理执行，主要职责包括：

- 根据配置和权重初始化推理引擎
- 调用模型对单帧样本进行预测
- 输出推理结果及统计信息

该模块是系统的核心计算模块。

#### 4.3.2 `runtime/visualizer.py`

该模块负责推理结果可视化，当前边界为：

- 输入：`sample + result`
- 输出：四宫格 RGB NumPy 图像

该设计使可视化逻辑与页面展示逻辑解耦。

#### 4.3.3 `runtime/exporter.py`

该模块负责导出演示结果，当前支持：

- 保存结果图像
- 保存单帧 JSON 结果
- 追加 CSV 摘要
- 通过 `export_all()` 串联完成完整导出

该模块不参与页面逻辑，只负责结果持久化。

#### 4.3.4 `runtime/validators.py`

该模块负责启动前校验与运行前样本校验，主要包括：

- 配置文件校验
- 权重文件校验
- `demo_data` 目录结构校验
- `index.json` 结构与路径有效性校验
- 输出目录校验
- 单帧样本结构校验

该模块仅做校验，不参与推理、可视化或导出。

#### 4.3.5 `runtime/logger.py`

该模块提供统一日志能力，主要包括：

- 控制台日志输出
- 文件日志输出到 `outputs/logs/app.log`
- 避免重复 handler
- 提供启动、校验、推理、导出等关键节点的日志接口

该模块采用最小可用设计，不引入复杂日志框架。

#### 4.3.6 `runtime/model_bootstrap.py`

该模块负责模型启动所需的底层准备工作，如权重路径解析、模型构建等，是推理引擎的支撑模块。

---

### 4.4 数据访问层

#### 4.4.1 `data_access/index_loader.py`

负责读取并解析 `demo_data/index.json`，将索引信息转换为系统可用的结构。

#### 4.4.2 `data_access/image_loader.py`

负责加载演示图像数据。

#### 4.4.3 `data_access/npz_loader.py`

负责加载演示样本的 `.npz` 特征文件。

#### 4.4.4 `data_access/sample_provider.py`

该模块是数据访问层的统一入口，主要职责包括：

- 列出演示样本 `frame_id`
- 根据 `frame_id` 读取对应样本
- 组装图像、特征、元信息等完整 sample 对象

它是运行主链中的第一个节点。

---

### 4.5 模型层

#### 4.5.1 `models/mome_model.py`

负责定义 MoME 模型结构，是系统推理能力的核心模型实现。

#### 4.5.2 `models/backbones.py`

负责定义模型使用的主干网络结构，为 `mome_model.py` 提供底层网络组件。

当前版本中，模型层仅服务于离线推理，不直接对外暴露页面交互接口。

---

### 4.6 配置与资源层

#### 4.6.1 配置文件

- `config/config.yaml`
- `config/deploy.yaml`
- `config/ui.yaml`

用于描述系统配置、部署参数、界面参数等。

#### 4.6.2 权重资源

- `weights/road_mome_v12_best.pth`

为当前离线演示主推理权重。

#### 4.6.3 演示数据资源

## 5. 数据流设计

当前版本采用单帧、同步、显式触发的数据流设计。其主要过程如下：

### 5.1 启动阶段数据流

1. 用户启动 `app.py`
2. `app.py` 调用 `app_bootstrap.initialize_runtime()`
3. `initialize_runtime()` 先执行 `run_startup_validation()`
4. 校验通过后，初始化：
   - `LocalSampleProvider`
   - `MomeInferEngine`
   - `ResultExporter`
5. 生成 `RuntimeContext`
6. `app.py` 创建页面布局并注册回调

### 5.2 运行阶段数据流

当用户在页面中选择 `frame_id` 并点击“运行检测”或“运行并导出”时，数据流如下：

1. UI 回调获取当前 `frame_id` 和阈值
2. `sample_provider.get_sample(frame_id)` 读取样本
3. `validate_runtime_sample(sample)` 执行运行前校验
4. `infer_engine.predict(sample, threshold=...)` 执行模型推理
5. `visualizer.draw_4panel_result(sample, result, save_path=None)` 生成四宫格图像
6. 若用户点击“运行并导出”，则调用 `exporter.export_all(sample, result, image_rgb)`
7. UI 回调将结果图、统计信息和状态信息更新到页面

该流程为同步串行流程，便于答辩展示和故障定位。

## 7. 校验与异常处理设计

### 7.1 启动前校验

启动前校验由 `runtime/validators.py` 提供，主要检查：

- 配置文件是否存在且可解析
- 权重文件是否存在
- `demo_data` 目录结构是否完整
- `index.json` 是否存在且非空
- 索引中的 `jpg` / `npz` 路径是否真实可达
- 输出目录是否存在或可创建、是否可写

若校验失败：

- 页面不直接崩溃
- 启动阶段停止继续初始化
- 状态区显示简洁错误信息
- 运行按钮保持禁用

### 7.2 运行前样本校验

在读取样本后、模型推理前，系统调用 `validate_runtime_sample(sample)`，检查：

- 关键字段是否存在
- 图像是否为空
- 特征矩阵形状是否符合要求
- `meta` 是否包含有效 patch

若校验失败：

- 不进入模型推理
- 页面显示“样本校验失败”
- 保持当前页面结构不崩溃

### 7.3 异常处理原则

当前版本异常处理遵循以下原则：

- 页面不直接显示 traceback
- 页面只展示简洁、用户可理解的错误摘要
- 代码层保留异常传播和异常类型区分能力
- 启动异常、样本异常、推理异常、可视化异常、导出异常分别处理

这种设计兼顾了用户友好性与工程可维护性。

## 8. 日志设计

日志系统由 `runtime/logger.py` 提供，采用 Python 标准库 `logging` 实现。

### 8.1 日志输出位置

- 控制台
- `outputs/logs/app.log`

### 8.2 日志记录内容

当前版本已覆盖以下关键节点：

- 应用初始化开始
- 启动校验通过 / 失败
- `LocalSampleProvider` 初始化成功 / 失败
- `MomeInferEngine` 初始化成功 / 失败
- `ResultExporter` 初始化成功 / 失败
- 运行检测开始
- 样本读取成功 / 失败
- 样本校验通过 / 失败
- 推理成功 / 失败
- 可视化成功 / 失败
- 导出成功 / 失败
- 校验失败摘要
- 输出目录或索引问题摘要

### 8.3 日志风格

日志采用人可读文本形式，包含：

- 时间
- 级别
- logger 名称
- 消息内容

例如：

- `INFO | 应用初始化开始`
- `INFO | startup validation 通过 | sample_count=...`
- `ERROR | 索引校验失败 | index_path=...`
- `INFO | 开始推理 | frame_id=...`
- `ERROR | 导出失败 | frame_id=...`

当前日志系统不引入远程日志、滚动日志或 JSON 结构化日志，符合演示平台“最小可用”的定位。

## 9. 当前版本边界与后续扩展方向

### 9.1 当前版本边界

当前版本能力边界明确如下：

#### 已实现

- 单机浏览器演示
- 单页 Dash 界面
- 单帧样本选择与检测
- 四宫格结果显示
- 导出图像 / JSON / CSV
- 启动前校验
- 运行前样本校验
- 统一日志记录

#### 未实现

- 真实硬件接入
- 实时视频流处理
- 批量样本处理
- 在线服务化部署
- 用户认证与权限管理
- 训练界面
- 任务队列与异步调度

因此，本系统当前更适合作为**课程设计演示平台、离线验证界面和答辩展示工具**，而不是完整的生产级在线系统。

### 9.2 后续扩展方向

在保持现有结构的基础上，后续可考虑扩展以下能力：

1. 配置管理增强  
   增加更细粒度的配置项解释和界面化配置展示。

2. 日志与监控增强  
   引入更完整的错误统计、运行统计和日志分级策略。

3. 导出能力增强  
   扩展导出格式，支持更丰富的实验记录。

4. 样本管理增强  
   支持更灵活的数据集组织方式和索引管理工具。

5. 多帧与批处理能力  
   在当前单帧链路稳定的前提下，进一步扩展为批处理或序列处理。

6. 服务化改造  
   在后续需要时，可将当前单机演示平台逐步演进为前后端分离的系统。

## 10. 结论

`road_mome_demo_v1` 当前版本围绕“单机浏览器演示平台”这一目标，已形成较清晰的模块分层：

- `app.py + app_bootstrap.py` 负责入口与装配
- `ui/` 负责页面布局与交互
- `runtime/` 负责推理、可视化、导出、校验和日志
- `data_access/` 负责演示样本读取
- `models/` 负责模型定义
- `config / weights / demo_data / outputs` 负责资源支撑

整体架构与当前实现一致，职责边界明确，适合用于课程设计答辩、演示说明和后续迭代开发。
